# 📌 03 - Document, Collection, and Database

## 1️⃣ What is a Database in MongoDB?

A database in MongoDB is a container for collections.

Structure hierarchy:

MongoDB Server  
→ Database  
→ Collection  
→ Document  
→ Fields  

Example:

Database: trainingDB

You can create multiple databases like:
- trainingDB
- productionDB
- analyticsDB

Each database is independent.

---

## 2️⃣ What is a Collection?

A collection is similar to a table in RDBMS.

- It stores multiple documents
- It does NOT enforce fixed schema
- Documents inside can have different structures

Example:

Collection: students

It contains student documents.

---

## 3️⃣ What is a Document?

A document is a single record stored in MongoDB.

It is stored in JSON-like format (BSON internally).

Example:

```json
{
  "name": "Bhanu",
  "age": 23,
  "cgpa": 8.5
}
````

Each document:

* Contains key-value pairs
* Has a unique `_id` field
* Can contain nested objects and arrays

---

## 4️⃣ Students Collection Example (Complete Hierarchy)

```
MongoDB Server
 └── trainingDB
      └── students
           ├── { name: "Bhanu", age: 23, cgpa: 8.5 }
           ├── { name: "Anil", age: 24, cgpa: 8.9 }
           └── { name: "Priya", age: 22, cgpa: 9.1 }
```

---

## 5️⃣ Understanding `_id` Field

Every document contains:

```json
"_id": ObjectId("65fa1234abcd123456789000")
```

### Why `_id` is Important:

* Primary key of the document
* Automatically indexed
* Ensures uniqueness
* Used internally for replication

You can also manually define `_id`:

```javascript
db.students.insertOne({
  _id: 101,
  name: "Bhanu",
  age: 23,
  cgpa: 8.5
});
```

---

## 6️⃣ Differences from RDBMS Structure

| Concept     | RDBMS    | MongoDB    |
| ----------- | -------- | ---------- |
| Database    | Database | Database   |
| Table       | Table    | Collection |
| Row         | Row      | Document   |
| Column      | Column   | Field      |
| Primary Key | id       | _id        |

---

## 7️⃣ Why This Matters in Production (SRE Perspective)

### Scenario: High Memory Usage

If a collection contains:

* Large documents
* Many unused fields
* Deep nesting

It increases:

* RAM usage
* Query latency
* Disk I/O

As SRE, you must:

* Understand document structure
* Identify poor schema design
* Suggest optimization

---

### Scenario: Duplicate Data Issue

If `_id` is manually set incorrectly:

* Duplicate key error
* Application insert failures
* 500 API errors

Root cause investigation:

* Check `_id` uniqueness
* Check indexing

---

## 8️⃣ Best Practices for Document Design

✅ Keep documents under 16MB
✅ Avoid unnecessary nesting
✅ Use embedding wisely
✅ Use consistent field naming
✅ Use indexes on frequently searched fields

---

## 9️⃣ Common Errors & Fixes

### ❌ Error: E11000 duplicate key error

Reason:

* Duplicate `_id`

Fix:

* Remove manual `_id`
* Or ensure unique value

---

### ❌ Error: Document too large

Reason:

* Exceeds 16MB limit

Fix:

* Break into smaller documents
* Use referencing instead of embedding
